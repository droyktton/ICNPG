Why You're Seeing "Not Supported"
The cudaDeviceCanAccessPeer check returns false if the hardware topology doesn't allow a direct path. This typically happens for a few reasons:


PCIe Topology: The GPUs might be connected to different CPU sockets or different PCIe Root Complexes that don't support peer-to-peer routing.

WDDM on Windows: If you are on Windows, GPUs often need to be in TCC (Tesla Compute Cluster) mode rather than WDDM mode to support P2P.


Hardware Generation: Older or consumer-grade cards sometimes lack the specific hardware links required for P2P.

Interpreting Your Results
In your matrix, the connection between GPU 0 and GPU 1 is labeled as NODE. According to your legend:

NODE: Connection traversing PCIe as well as the interconnect between PCIe Host Bridges within a NUMA node.

This means that while both GPUs are technically in the same NUMA node (Node 0), they are connected to different PCIe Host Bridges. For P2P to work over PCIe, GPUs typically need a more direct path, such as PIX (at most a single bridge) or PXB (multiple bridges without traversing the Host Bridge).

Because your traffic has to traverse the interconnect between host bridges, the hardware does not support the direct routing required for Peer-to-Peer memory access.

################################
Target Labels for P2P Support
If your hardware is optimized for multi-GPU communication, you should see the following acronyms instead of NODE:


NV# (e.g., NV1, NV12): This indicates a connection via NVLink, which is the gold standard. P2P is almost certainly supported and will provide significantly higher bandwidth than any PCIe-based connection.


PIX: This stands for a connection traversing at most a single PCIe bridge. This is a very direct path that typically supports P2P.


PXB: This indicates the connection traverses multiple PCIe bridges but does not go through the PCIe Host Bridge (CPU). This path usually supports P2P access.

The Next Lesson: Fallback with Pinned Memory
Since P2P is unavailable, the next step in teaching Multi-GPU CUDA is showing how to optimize the "Fallback" path (GPU $\rightarrow$ Host $\rightarrow$ GPU). 
The standard method of moving data through the CPU is slow because it uses "Pageable" host memory. 
To make this faster, you should teach Pinned (Page-Locked) Memory.

Why Pinned Memory?
Direct Memory Access (DMA): The GPU can copy data directly from Pinned Host Memory without the CPU's help, providing higher bandwidth.
Asynchronous Transfers: You can only overlap data transfers with kernel execution (using CUDA Streams) if you use Pinned Memory.
